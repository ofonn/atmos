Active bugs — confirmed from screenshots
Bug · CriticalRefresh icon disappears instead of animating

When the Daily Briefing refresh button is tapped, the icon vanishes entirely from the DOM instead of rotating in place. This leaves a blank space and gives no feedback that anything is happening. The icon should persist and spin for the duration of the refresh, then stop. Disappearing is worse than no animation at all — it reads as a broken state.

Fix
Keep the icon element mounted at all times. On refresh trigger: add a CSS class that applies `animation: spin 1s linear infinite`. On completion: remove the class. Never toggle `display:none` or unmount the icon. Use a single `@keyframes spin { to { transform: rotate(360deg); } }` rule.
Bug · CriticalHourly strip cards use hardcoded times (16:00, 19:00, 22:00)

The hourly forecast strip always starts at 16:00, 19:00, 22:00, 01:00 regardless of the current time. If it's 5:00 PM, the first future slot should be 7:00 PM. If it's 11:00 PM, it should be 1:00 AM. The "Now" slot is correct — but everything after it is hardcoded, not driven by the current time. This is a data logic bug, not a UI bug.

Fix
Calculate slots dynamically: `startHour = Math.ceil(currentHour / 3) * 3`, then generate slots at +3h intervals from there. If the interval is not 3h, derive it from the API data — use the actual forecast timestamps and filter to show the next N hours from `Date.now()`. Never hardcode 16, 19, 22, 1 as time values.
Bug · Critical60-minute precipitation chart not rendering

The 60-Min Precipitation section on the Conditions screen shows "No precipitation expected in the next hour" with a timeline row (Now, +15m, +30m, +45m) but no actual chart visualization appears. The section exists and the label works, but the chart itself is empty — no bars, no lines, no visual data. This is a rendering failure.

Fix
Audit the chart initialization code — likely a missing canvas reference, a failed API response, or a data shape mismatch. The chart should render even for zero-precipitation data (flat line at 0mm). Test with both zero and non-zero precipitation data. Use Open-Meteo's `precipitation` field at 15-minute intervals if available, or interpolate from hourly data.
Bug · Critical24h → 12h time format toggle has no effect

Switching between 24h and 12h in Settings changes the UI state of the segmented control, but no time values anywhere in the app actually reformat. Hourly strip slots still show 16:00, 19:00, 22:00. AI response timestamps still use 24h. The setting is persisted nowhere and applied nowhere.

Fix
Store the time format preference in localStorage (key: `atmos_time_format`). Create a global `formatTime(date, format)` utility function that all time displays consume. On settings change, re-render all visible time displays. On app load, read the preference and apply it before first render. Affected surfaces: hourly strip, AI response timestamps, Proactive Insight timestamp, Conditions updated-at label, chat message timestamps.

In the Whispering Pines screenshot, the temperature is -1°C but the editorial headline reads "It's super hot and a bit grey out." New improvements not in previous critiques
ArchitectureMigrate all weather data to Open-Meteo exclusively

The current app appears to use a mixed or undocumented weather data source. Open-Meteo is free, requires no API key, has excellent global coverage, provides hourly and 15-minute data, includes precipitation probability, UV index, wind, air quality, and supports historical data. It should be the single source of truth for all weather data in the app.

Implementation plan
1. Audit every weather data fetch in the codebase. 2. Create a single `weatherService.ts` module that wraps all Open-Meteo API calls. 3. Map Open-Meteo's field names to the app's internal data model. 4. Key endpoints needed: forecast (hourly + 15min), air quality, geocoding. 5. Base URL: `https://api.open-meteo.com/v1/forecast`. 6. Add `&timezone=auto` to all calls. 7. Remove all other weather API dependencies and their API keys/credits. 8. Test every screen against the new data source before going live.

None of the screenshots show any severe weather alert UI — no red banner, no alert icon, no emergency state design. For users in regions with extreme weather (heat advisories, storms, flooding), this is a safety gap. The app needs an alert layer that overrides the calm visual language when conditions are genuinely dangerous.

Red background, white text, warning icon, brief description, and a "View details" CTA. This banner should be the first thing visible on Home — above the temperature — and should not be dismissible until the alert expires. Test with a mock alert before connecting to live data.
